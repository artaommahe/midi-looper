# MIDI Looper iPhone App V1 Architecture

Related document: `docs/specs/2026-04-04-v1-product-spec.md`

## Purpose

This document describes the intended runtime architecture for v1 of the MIDI looper app.

The main goal is not architectural completeness. The goal is to define a simple structure that supports:
- correct MIDI behavior
- stable timing on a real iPhone device
- low-friction live control
- small, incremental implementation steps

This document should help future implementation work stay aligned on responsibilities, timing behavior, and safety rules.

## Scope

This architecture covers v1 behavior defined by the active product spec, including:
- MIDI input from a Roland FP-10 over USB
- MIDI output back to the FP-10
- live MIDI thru
- 4 loop tracks
- first loop defines the master loop length
- tempo is derived from the first completed loop
- quantized record start and end
- overdub by layering MIDI events
- mute, solo, and clear on loop boundaries
- global play and stop
- visual beat indication

This document does not define future features such as:
- session persistence
- audio playback or audio looping
- advanced MIDI editing
- external sync support
- AUv3 or plugin hosting

## System Overview

At a high level, the app should behave like this:

`FP-10 -> CoreMIDI Input -> Looper Engine -> CoreMIDI Output -> FP-10`

The app has four main runtime areas:
- MIDI I/O layer
- looper engine
- track data and track state
- UI layer

The main design rule is that timing-critical behavior should live in the engine side, not in SwiftUI view logic.

## Core Components

### MIDI I/O Layer

Responsibilities:
- discover available MIDI endpoints
- connect to the selected MIDI input and output
- receive incoming MIDI events from the FP-10
- immediately pass live MIDI events through to the output
- send playback events generated by the looper engine
- surface device connection state to the app

Non-responsibilities:
- loop state decisions
- record and overdub policy
- UI-specific behavior

This layer should stay as thin as practical. It is responsible for reliable event transport, not product rules.

### Looper Engine

Responsibilities:
- own the transport state
- own the master clock
- determine loop boundaries
- decide when queued actions become active
- route record, playback, overdub, mute, solo, and stop behavior
- schedule playback events back to MIDI output
- enforce note-off and pedal safety rules

Non-responsibilities:
- direct rendering of UI
- persistence in v1

The looper engine is the behavioral core of the app.

### Track Model

Each track should own only the data needed for loop playback and control.

At minimum, each track needs:
- an identifier
- current visible state
- recorded MIDI events for the loop
- queued action for the next boundary, if any
- active note tracking for note-off safety
- playback eligibility based on mute and solo state

The exact storage format can evolve, but the model should stay simple and focused on live playback behavior.

### UI Layer

Responsibilities:
- render transport state
- render beat or bar feedback
- render track state and queued actions
- send user intents such as record, mute, solo, clear, play, and stop

Non-responsibilities:
- timing calculations
- loop scheduling
- direct note safety decisions

The UI should observe engine state, not define timing behavior.

## Timing Model

## Master Clock

The app should maintain a single master clock for loop scheduling.

This clock should be the source of truth for:
- current transport position
- current beat or bar indication
- loop boundary timing
- quantized action activation

UI rendering should consume clock state, but should not be the clock source.

## Master Loop Length

The first completed loop defines the session's master loop length.

In v1, the first completed loop also defines the session tempo.

After that:
- all other tracks use the same loop length
- track playback wraps on the same master boundary
- overdubs are recorded relative to that loop length

V1 should not support independent track lengths.

Before the first loop is completed:
- the app may show a placeholder or pending tempo state
- the app should not depend on a manually configured transport tempo

## Quantization Rules

V1 uses quantized loop start and loop end behavior.

That means:
- when a track is armed to record, actual recording begins at the next loop boundary
- when recording is stopped, the track closes on the next loop boundary
- mute, solo, and clear state changes also apply on the next loop boundary

Within the loop:
- note timing should remain unquantized
- recorded event timing should reflect the actual performed offsets inside the loop

This keeps the live feel while making loop control predictable.

## Transport Model

Transport should at least support:
- stopped
- running

When stopped:
- loop playback does not continue
- any currently active playback notes must be safely released

When running:
- live MIDI thru remains active
- active tracks play according to their loop content and state

## Track State Model

Visible track states in v1 should stay simple.

Primary states:
- empty
- armed
- recording
- playing

Control overlays:
- muted
- soloed

Muted and soloed should be modeled as overlays on top of a primary state, not as standalone primary states.

Examples:
- a track can be `playing` with a `muted` overlay
- a track can be `playing` with a `soloed` overlay
- a track should not be forced into a separate standalone `muted` state if `playing + muted overlay` expresses the behavior more directly

Internally, the engine may need more precise flags than the UI shows. That is acceptable as long as the visible behavior remains simple and predictable.

## Suggested State Transitions

### Empty Track

- `Rec/OD` arms the track for recording at the next boundary
- `Clear` has no effect
- `Mute` has no effect
- `Solo` has no effect unless later behavior requires it

### Armed Track

- next boundary changes state to `recording`
- cancelling before the boundary returns the track to `empty`

### Recording Track

- next record-stop action closes at the next boundary
- if this is the first completed loop, it defines the master loop length
- after close, the track enters `playing`

### Playing Track

- `Rec/OD` queues overdub recording
- `Mute` queues muted playback state for next boundary
- `Solo` queues solo state for next boundary
- `Clear` queues loop removal for the next boundary and then returns the track to `empty`

### Playing Track With Muted Overlay

- remains synchronized to the master loop while not sounding
- unmute should restore playback on a future boundary

### Playing Track With Soloed Overlay

- playback eligibility of all tracks must be recalculated when solo is active
- non-soloed tracks should be suppressed safely at the boundary

## MIDI Event Model

V1 should record the minimum useful set of MIDI event types:
- note on
- note off
- sustain pedal control changes

Additional event types can be ignored unless the product spec expands.

Each recorded event should be stored with:
- track ownership
- time offset relative to loop start
- raw MIDI message data or an equivalent compact representation

The model should be simple enough to replay events accurately and reason about note safety.

## Boundary Action Rules

Boundary behavior should be explicit.

In v1:
- record start applies at the next loop boundary
- record stop applies at the next loop boundary
- overdub activation applies at the next loop boundary
- mute applies at the next loop boundary
- solo applies at the next loop boundary
- clear applies at the next loop boundary

The user should be able to see when an action has been queued for the next boundary.

## Note-Off Safety

This is a critical part of the architecture.

The app must not leave hanging notes or stuck sustain states when playback state changes.

The engine should explicitly track sounding notes per track so it can safely release them when needed.

Safety rules:
- stopping transport must send required note-off events for playback-generated notes
- muting a track must not leave playback notes hanging
- solo transitions must release notes from tracks that become inactive
- clearing a track must not leave notes active
- sustain pedal state must be released safely when playback or track eligibility changes require it

Exact implementation details can evolve, but this behavior is not optional.

## UI Architecture

V1 should use a single-screen pedalboard layout.

Top area:
- play and stop controls
- transport state
- tempo display derived from the first completed loop
- beat or bar indicator

Main area:
- 4 large track panels

Each track panel should show:
- track identity
- current state
- queued action if present
- Rec/OD
- Mute
- Solo
- Clear

The UI should prioritize fast recognition and fast tapping over information density.

## Device And Error Handling

The app should handle the following situations without unclear behavior:
- no MIDI device connected
- input connected but output unavailable
- device disconnect during playback
- device reconnect after disconnect

At minimum, the app should surface enough state for the user to understand whether MIDI input and output are available.

If the active device disconnects during playback:
- transport should fail safely
- playback-generated notes should be considered released locally
- UI should clearly reflect disconnected state

## Validation Strategy

Every meaningful implementation step should be validated on a real iPhone with the FP-10.

Minimum validation checks:
- MIDI input is detected correctly
- MIDI output is detected correctly
- note-on and note-off pass through correctly
- sustain pedal passes through correctly
- live thru feels immediate enough for normal play
- first loop correctly defines master loop length
- overdub playback stays aligned to loop boundaries
- mute and solo changes occur on expected boundaries
- no stuck notes remain after stop, mute, solo, or clear
- loop playback remains stable over several minutes

## Open Questions

These are acceptable to leave open until implementation pressure requires a decision:
- exact internal representation for MIDI events
- how much UI feedback should be shown for queued actions in the earliest prototype

When these decisions affect behavior, they should be resolved in code and reflected in the active spec or architecture notes.
